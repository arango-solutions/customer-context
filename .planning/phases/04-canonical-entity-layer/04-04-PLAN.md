<objective>
Close the remaining BLOCKER on Success Criterion #3: the demo-critical integrity/coref gate is a false guarantee. A live run prints `Demo-critical accuracy: 3/3 = 100% — PASS` and exits 0 while only 3 of the 9 demo-critical entities are examined and the join key is wrong for the rest.

Two confirmed defects (from 04-VERIFICATION.md):
- **CR-01 — coverage hole.** Both `verify_coref_eval.py` (lines 356-364) and `verify_entity_bridge.check_demo_critical` (lines 413-454) compute the denominator from what is *present* (ground-truth-covered expected ids; hubs that happen to exist) rather than from the authoritative 9-id set. The 6 ids not covered (Sarah Chen, Michael Torres, Meridian Logistics, Northwind Analytics, Enterprise 2026 contract, ArangoGraph 2026 contract) never enter the denominator — an absent or mis-linked one is a free pass.
- **CR-02 — wrong join key.** `verify_coref_eval.py` (lines 208-219) filters `e.entity_name IN @mentions`, where `@mentions` are typo/role surface forms ("Ockafor", "JO", "the director"). AutoGraph stores entities under their resolved canonical `entity_name`, so these never match — the gate measures "is this typo stored as an entity_name," not "did this mention resolve to the correct canonical id."

The signal both gates must use is the **`same_as` bridge path**, not the KG-side `entity_id` stamp. The builder's `_upsert_structured_edges` writes a structured-side `same_as` edge (Contact/Account/Contract → `canonical_entities/{entity_id}` hub) for every one of the 9 demo-critical ids — so a hub reachable via `same_as` whose `canonical_id == expected_id` exists for all 9 on a correct build. The `stamp_entity_ids` step, by contrast, only stamps `customer360_Entities` rows AutoGraph actually extracted (~3 of 9); the other 6 have NO KG row to stamp, ever. A stamp-keyed gate over a denominator of 9 is therefore *permanently red on a correct bridge* — the mirror image of the original false-PASS. Task 2 already gates on the structured-side `same_as` signal precisely for this reason; Task 3 must use the same all-9-reachable signal rather than reverting to the stamp.

This plan makes the gate able to fail when the bridge is wrong — and able to pass when it is right: denominator = 9 in BOTH gates, an absent/unlinked/mis-linked id is a FAILURE, coref measured through the `same_as` bridge path (mention → bridge resolution → hub whose `canonical_id == expected_id`, with the edge's `matched_surface_form`/`match_method` checked against ground truth for the 3 covered ids), coverage status surfaced for the 6 uncovered ids, and the 9-id set consolidated into one shared module so the two gates can never drift (IN-03).

Purpose: SC#3 requires the integrity/coref validation to actually be able to fail — without becoming a permanent false-negative red that nobody trusts. Today it ships green over 3 of 9 with a structurally wrong numerator. This is the project's explicitly-forbidden failure mode — a confident PASS over a thing that was never validated.
Output: A shared `scripts/demo_critical.py`; corrected `verify_coref_eval.py` and `verify_entity_bridge.py` gates; the builder importing the shared set; and a negative + positive test proving both gates exit 1 when a demo-critical id is unlinked/mis-linked and exit 0 on a fully-built bridge.

This plan does NOT re-touch the already-closed no-double-resolution work (`_safe_alias_set` / `_safe_name_to_id_set` in build_entity_bridge.py raise ValueError before any DB write — verified genuinely fixed by 04-03).

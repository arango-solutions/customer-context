<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Vercel Cron → /api/canary | Internal-origin scheduled request; bypasses SSO, must be self-gated by CRON_SECRET |
| public/browser → /api/canary | Untrusted; must be rejected (401) without the bearer; SSO still applies to humans |
| canary → ArangoDB + OpenAI | New env-reading surface (reads ARANGO_*/OPENAI_API_KEY), unlike the secret-free /api/health |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-04 | Denial of Service | unauthenticated canary triggers expensive agent runs (cost DoS) | mitigate | `CRON_SECRET` bearer guard; bare 401 without it; the only caller is Vercel Cron (auto-injected bearer) + presenter curl. |
| T-07-05 | Information Disclosure | canary leaks DB/agent error, stack, or secret on failure | mitigate | Generic `{status:'red', ms}` 503; no error body, no stack, no secret serialized (mirrors T-06-07). |
| T-07-06 | Information Disclosure | NEW env-reading surface widens disclosure vs. secret-free /api/health | mitigate | Separate `/api/canary` route (health stays secret-free, T-06-01 preserved); never serialize ARANGO_*/OPENAI_API_KEY into the response; documented as a new surface here. |
| T-07-07 | Spoofing | cron treated as a browser request / SSO challenge instead of authenticated | mitigate | Cron originates inside Vercel (bypasses Deployment Protection); guard validates `Bearer $CRON_SECRET`; verified via `vercel logs` 200 (Pitfall 4). |
| AR-06-2 | Denial of Service | `/api/ask` per-IP rate-limit | accept (carried forward, D-09) | Still deferred in Phase 7; SSO + presenter-driven access = no public surface to hammer; re-open only on public custom domain. |
</threat_model>
